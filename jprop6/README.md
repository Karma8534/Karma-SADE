# jprop6

## What jprop6 Is
`jprop6` is a forensic reset of the Nexus plan into a foundation-first execution rail.
It is not a cosmetic rewrite and not a promise that the full system is complete.
It is a hardened plan + minimal Electron harness scaffold + binary proof gates.

## Why the Old Nexus Plan Failed
The old plan mixed three incompatible modes in one document:
1. Runtime claim ledger (`shipped/live/done`) that was not consistently re-verified.
2. Architecture aspiration (future patterns, multi-agent expansions, transport growth).
3. Active implementation order.

That mixing allowed phase advancement while foundational continuity and runtime truth were still unstable.

## Foundation vs Non-Foundation
Foundation in `jprop6` means only what must be true before growth:
1. Canonical runtime truth rails (claim != proof).
2. Continuity substrate (session envelope, recovery contract, explicit failure handling).
3. Browser/Electron shared session contract.
4. Minimal Electron harness scaffold that honestly embeds `https://hub.arknexus.net` and exposes only minimal continuity seams.

Non-foundation (explicitly deferred):
1. Voice stack expansion.
2. Plugin ecosystem growth.
3. Multi-transport expansion beyond browser + Electron.
4. Broad orchestrator fantasies without proven continuity contracts.

## Minimal Electron Harness Definition
The scaffold in `MINIMAL_HARNESS/` proves:
1. Electron shell wiring with secure preload boundary.
2. Hub load path to `https://hub.arknexus.net`.
3. Local continuity envelope read/write seam.
4. Runtime self-description APIs.

It does **not** prove:
1. Full merged workspace UX completion.
2. End-to-end multi-agent orchestration.
3. Full continuity reliability in production.
4. Any watcher autonomy claim.

## How to Execute jprop6 (Phase-Locked)
1. Run Phase 0 gates only.
2. If any Phase 0 gate fails, stop and fix before Phase 1.
3. Repeat gate discipline for every phase.
4. Run simulation and break tests after each phase.
5. Never carry a failed dependency into a later phase.

Binary gates: [`EXECUTION_GATES.md`](./EXECUTION_GATES.md)

## Validation Model
Each phase requires:
1. Objective/scope lock.
2. Deliverable evidence.
3. Proof command evidence.
4. Break attack outcomes.
5. Explicit PASS/FAIL.

No “mostly done.”

## Intentionally Deferred
Deferred until foundation is certified:
1. New plugin classes.
2. New watcher responsibilities.
3. New transport channels.
4. Advanced agent autonomy layers.

Those appear only after `GROUND-TRUTH VERIFIED 100% = YES` in `PROOF.md`.
