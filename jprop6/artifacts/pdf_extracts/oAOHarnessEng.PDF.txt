System design
Before (Engineers): Engineers define the core

After (Engineers): Still engineer-led, but with

Architecture decisions
Before (Engineers): Engineers decide patterns and

After (Engineers): Still engineer-led, but more

Writing code
Before (Engineers): Engineers implement features

functions, wiring dependencies, and making

After (Agents): Agents generate implementation

Code review
Before (Engineers): Engineers review logic, style,
correctness, and maintainability, and then
request changes.

===== PAGE 11/60 =====
After (Engineers): Engineers still review, but

Debugging failures
Before (Engineers): Engineers reproduce issues,
inspect logs and traces, isolate root causes, and

After (Engineers + Agents): Engineers and agents

Before (Engineers): Engineers author unit,

Refactoring
Before (Engineers): Engineers restructure code for

After (Agents): Agents perform much of the

Engineers define the target architecture, write

OpenAI harness engineering: Production Outputs
Harness Engineering Pattern 1 (from
OpenAI’s experiment): Repository as Sole
Source of Truth
Here is the first principle that makes zero-manual-
code possible, and it is brutally simple.
If it is not in the repository, the agent does not
know it exists.


===== PAGE 16/60 =====
As Ryan Lopopolo wrote in OpenAI’s “Harness
engineering” post: “From the agent’s point of view,
anything it can’t access in-repo effectively doesn’t
exist, so OpenAI pushed more and more context
into the repository over time.”
This sounds obvious. It is not. Think about how
much knowledge lives outside your codebase right
now. Architecture decisions in Confluence pages.
API contracts in Notion docs. Deployment
procedures in Slack threads. Onboarding
knowledge in someone’s head.
An agent cannot read your Slack. An agent cannot
attend your standup. An agent cannot absorb the
tribal knowledge that your senior engineer carries
around. The only reality an agent experiences is
the repository it operates in.
OpenAI made this explicit and absolute. Every
piece of information an agent could possibly need
to write correct code had to exist inside the

===== PAGE 17/60 =====
repository. If a convention was not documented in
the repo, the agent would violate it. If an API
contract was not specified in the repo, the agent
would guess wrong. If an architectural boundary
was not encoded in the repo, the agent would cross
it.
This is not a documentation initiative. This is a
survival requirement.
Why This Harness Pattern Matters at
Scale
Consider the failure mode without this pattern. At
200,000 lines per month, an agent that consistently
applies the wrong naming convention will apply it
thousands of times before anyone catches it. An
agent that misunderstands a module boundary will
cross it in dozens of files. The damage scales with
the throughput.
The repository-as-truth pattern converts
documentation from “nice to have” to “load-

===== PAGE 18/60 =====
bearing structure.” Every missing convention is a
defect multiplier. Every undocumented boundary
is a source of compounding errors.
OpenAI found, predictably, that the first two
months of the experiment required heavy
documentation investment before throughput
could scale. The harness had to be built before the
agents could run.
What Goes Into the Repo
The team categorized repository knowledge into
three tiers:


===== PAGE 19/60 =====
Foundational (monthly): Architecture decisions,
module boundaries, coding standards, naming
conventions
Operational (weekly): API contracts, data
schemas, deployment configs, environment
setup
Tactical (daily): Task-specific context, current
sprint goals, known issues, workarounds
Traditional teams can get away with sloppy
documentation because humans fill in the gaps
with institutional knowledge. Agent teams cannot.
Every gap in repository documentation becomes a
gap in agent capability.
If you have ever onboarded a new hire and thought
“they will just pick this up from the team,”
congratulations: you have identified exactly the
kind of knowledge that will break your agents.
The boundary between agent-visible and agent-
invisible knowledge is absolute. There is no “well,

===== PAGE 20/60 =====
the agent can probably figure it out.” Either the
knowledge is in the repo or it does not exist.

Disclosure via Structured Directories
Repository-as-truth creates a new problem. If you
dump every piece of knowledge into a flat
documentation folder, the agent drowns. Context
windows are large but not infinite. Relevance
matters.
OpenAI solved this with what Ryan’s post calls
progressive disclosure: agents start with a small,
stable entry point and are taught where to look
next, rather than being overwhelmed up front.
Mechanically, this means a structured directory
layout paired with a brief map file (AGENTS.md in
OpenAI’s case) that acts as a table of contents.
The directory structure is not arbitrary. It follows a
deliberate hierarchy that mirrors how agents
navigate codebases. Agents start broad and go
deep. The structure enables that navigation.

in Practice

Harness Engineering Pattern 3 (from
OpenAI’s experiment): Application
Legibility
Here is where OpenAI’s harness engineering gets
genuinely interesting.

===== PAGE 27/60 =====
Application legibility means that every running
instance of the application is fully inspectable,
fully observable, and fully isolated. The agent does
not just write code. It boots the application, tests it,
observes it, and diagnoses it.
OpenAI harness engineering - Application Legibility
This required three capabilities that most
development environments do not have.
Isolated Bootable Instances
OpenAI made the application bootable per git
worktree, so Codex could launch one instance per


===== PAGE 28/60 =====
change, each with its own logs and metrics. Every
agent session gets its own complete instance of the
application. Not a shared staging environment. Not
a mock. A real, running instance that the agent
controls entirely.
Why does isolation matter at this scale? Consider a
scenario where 10 agents work simultaneously on
10 different features. Without isolation, Agent A’s
half-complete database migration breaks Agent B’s
test run. Agent C’s configuration change causes
Agent D’s service to fail in a way that has nothing
to do with Agent D’s code. Every agent wastes
cycles debugging problems created by other
agents.
With isolated instances, each agent operates in a
clean environment. Its failures are its own. Its
successes are reproducible. The agents can run in
parallel without interference.
Browser Tools

===== PAGE 29/60 =====
Agents do not just compile and run tests. They
interact with the application the way a user would.
OpenAI wired the Chrome DevTools Protocol into
the agent runtime so Codex could drive the UI and
reason about DOM snapshots and screenshots -
navigating the UI, clicking buttons, filling forms,
and observing visual output.
This matters because many bugs are invisible to
unit tests. A function that returns correct data but
renders it in the wrong place will pass every
integration test but fail every user interaction.
Agents with browser tools catch these failures
before merge, not after deployment.
The practical implication: browser tool access
transformed agents from “code writers that hope
the code works” to “code writers that verify the
code works from the user’s perspective.”
Full Observability Stack
OpenAI set up a local observability stack per
worktree, with LogQL/PromQL access for logs and

===== PAGE 30/60 =====
metrics. Every agent session included structured
logging, metrics collection, and trace visualization.
When an agent’s code caused an error, the agent
could read the stack trace, examine the logs, and
trace the request through the system.
This is not luxury infrastructure. Without
observability, an agent that generates buggy code
enters a debugging loop without information. It
guesses, modifies, reruns, and guesses again. With
observability, it reads the error trace, identifies the
line, understands the cause, and fixes the right
thing.
The combination of these three capabilities means
that agents operate in an environment as rich as
what a human developer experiences. They can
write code, run it, see what happens, read the
errors, and fix the problems. That tight feedback
loop is what makes autonomous code generation
viable at production scale.

===== PAGE 31/60 =====
Without application legibility, agents write code
blind. With legibility, they write, test, observe, and
iterate, just like a human developer would.

Harness Engineering Pattern 4 (from
OpenAI’s experiment): Mechanical
Architecture Enforcement
Trust but verify is good advice for hiring. It is
terrible advice for agents.
OpenAI’s fourth harness engineering pattern is
mechanical enforcement of architectural

===== PAGE 32/60 =====
constraints - a rigid architectural model per
business domain with strictly validated
dependency directions, enforced via custom
linters and structural tests. Not guidelines. Not
suggestions. Not comments in a
CONVENTIONS.md file that agents might read and
might follow. Actual linters that run automatically
and block merges when violated.
OpenAI harness engineering: Mechanical Enforcement
Here is the key insight: linters must include
remediation instructions.


===== PAGE 33/60 =====
A traditional linter says “naming convention
violated on line 47.” That is useful for a human who
knows the naming convention and just made a
typo. It is useless for an agent that has never seen
the convention and does not know what the
correct name should be.
What Actionable Linter Output Looks Like
OpenAI’s custom linters output structured
feedback designed for agent consumption:
ERROR: Naming convention violation
  File: modules/billing/src/invoice_handler.py
  Line: 47
  Found: `processInvoice`
  Expected pattern: snake_case function names
  Fix: Rename to `process_invoice`
  Reference: See modules/billing/CONVENTIONS.md#namin
That structured error message is not just
informative. It is actionable. The agent reads it,
understands exactly what went wrong, and fixes it
without human intervention. The linter becomes a
feedback loop, not just a gate.

===== PAGE 34/60 =====
Without the “Fix” line, the agent knows something
is wrong but must infer the correction. With the
“Fix” line, the agent has an instruction. The
difference in first-try success rates is significant.
What Gets Mechanically Enforced
OpenAI used four categories of automated
enforcement:
Naming conventions
What it checks: Function, class, variable names
match module patterns
Agent response: Auto-rename
Architecture boundaries
What it checks: No cross-module imports that
violate dependency rules
Agent response: Restructure imports
API contract compliance

===== PAGE 35/60 =====
What it checks: Endpoints match OpenAPI spec,
request/response schemas validated
Agent response: Fix schemas
Test coverage thresholds
What it checks: New code must include tests,
minimum coverage per module
Agent response: Generate missing tests
OpenAI harness engineering: Automated Enforcement
Each category includes machine-readable
remediation instructions. The agent runs the


===== PAGE 36/60 =====
linter, reads the failures, applies the fixes, and
runs the linter again. This loop continues until all
checks pass.
The humans never see the violations. They only
see the clean result. That is the whole point.
This is CI/CD principles applied to agentic
development workflows. The same logic that
makes automated test suites valuable for humans
makes automated linters with remediation
instructions valuable for agents. The mechanism is
the same. The consumer of the feedback is
different.

Harness Engineering Pattern 5 (from
OpenAI’s experiment): Throughput
Changes Merge Philosophy
This is the pattern that surprised me most, and it is
the one most teams will resist.

OpenAI harness engineering: Througput Merge Philosophy
In traditional development, blocking quality gates
make sense. You have a finite number of
developers. Each pull request gets careful review.
Merging is a deliberate, gated act. Speed matters,
but correctness matters more, because fixing a bad
merge is expensive.
When you have dozens of agents producing
hundreds of pull requests per day, that math
changes completely.
The Throughput Problem at Agentic Scale


===== PAGE 39/60 =====
Consider the arithmetic. A team of 5 developers
can review approximately 15 to 25 PRs per day with
reasonable thoroughness. That is the human
review ceiling.
An agentic workflow with 20 concurrent agents
can produce 40 to 80 PRs per day. You cannot scale
human review to match that rate. If every PR waits
for a human reviewer, you have recreated the exact
constraint you were trying to eliminate. The agents
can write code in parallel all day long, but if merge
is sequential and human-gated, your throughput is
limited by the speed of human review.
The New Merge Philosophy
OpenAI shifted to a fundamentally different
approach.
Merge fast. Detect fast. Fix fast.
OpenAI describes minimal blocking merge gates
and short-lived PRs. This article generalizes that

===== PAGE 40/60 =====
into a “merge fast, detect fast, fix fast” philosophy
for high-throughput agent workflows.
Instead of trying to prevent every possible problem
before merge, they built systems to detect and
remediate problems after merge. The linters still
run pre-merge. The tests still run pre-merge. But
the review process changed from “block until a
human approves” to “merge if all automated
checks pass, flag for human review async.”
Why This Works (and Why It Only Works with the
Other Harness Patterns)
This philosophy only holds because of the previous
four patterns. The repo has complete
documentation (Pattern 1). The structure is
navigable (Pattern 2). The application is fully
observable (Pattern 3). Architectural constraints
are mechanically enforced (Pattern 4). With all
four in place, the risk of a bad automated merge is
low enough that speed wins over caution.

===== PAGE 41/60 =====
Remove any one of those patterns and the fast-
merge philosophy becomes a liability. Agents
merging fast without observability means bad code
reaches production before anyone knows it is bad.
Agents merging fast without architectural
enforcement means module boundaries erode at
high speed.
The patterns are interdependent. They are a
system.
This does not mean abandoning quality. It means
moving quality enforcement from human gates to
automated systems. The standards do not drop.
The mechanism for enforcing them changes.

Practical Takeaways
If you are building agentic development
workflows, here are five concrete actions to take
this week.
1. Audit your repo for invisible knowledge. Walk
through your onboarding docs. Everything a new
hire needs to learn should be in the repository. If it
is in Slack, Confluence, or someone’s head, move it
to the repo. Quantify the gap: how many things
would break if you hired an agent today with only
your repo as context?
2. Create a brief map file. Start with a single
BRIEF.md at the root. List every module, its
purpose, its current state, and where to find
deeper documentation. Keep it under 200 lines. If
you cannot summarize your codebase in 200 lines,
the problem is not the line limit.
3. Add remediation instructions to your linters.
For every lint rule, add a “Fix” line that tells the

===== PAGE 48/60 =====
agent exactly what to do. “Naming violation” is not
enough. “Rename processInvoice to
process_invoice per
modules/billing/CONVENTIONS.md#naming" is
what agents need. The time investment here pays
back immediately in reduced linter iteration
cycles.
4. Set up isolated dev instances. Even if you start
with Docker Compose, give each agent session its
own environment. Shared environments create
race conditions that waste agent cycles and
produce confusing failures. Isolation is table stakes
for parallel agent work.
5. Rethink your merge gates. If automated checks
are comprehensive enough, consider reducing
blocking human review to high-risk changes only.
Let agents merge low-risk, well-tested changes
automatically. Track the percentage of agent PRs
that require human intervention. Drive that

===== PAGE 49/60 =====
number down by improving the harness, not by
adding more human review.
